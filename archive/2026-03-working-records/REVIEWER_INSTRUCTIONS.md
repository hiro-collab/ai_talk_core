# Reviewer Instructions

このプロジェクトでは、レビュー結果の記録先を用途別に分けています。

## 記録先

- `REVIEW.md`
  レビュー結果の主記録先です。Findings, open questions, 改善提案を書いてください。
- `DESIGN_REVIEW.md`
  UI / UX / 情報設計 / 見た目に関するレビューの主記録先です。デザインレビュー担当はこちらを使ってください。
- `SHARE_NOTE.md`
  合意済みの次アクションだけを書いてください。未確定の所見は書かないでください。
- `LOG.md`
  実行した確認コマンドや事実結果だけを書いてください。レビュー意見そのものは書かないでください。
- `MEMORY.md`
  長期的な設計前提や運用ルールとして確定したことだけを書いてください。

## 運用ファイルの注意

- `AGENTS.md` や `/init` 相当の初期化・運用ファイルは、重複作成しないでください
- 既存ファイルがある場合はそれを前提にし、別名ファイルや重複ファイルを増やさないでください
- 追加が必要と判断した場合は、まず提案してからにしてください

## 期待するレビュー形式

以下の順でまとめてください。

1. Findings
2. Open questions / assumptions
3. Recommended next actions

デザインレビュー担当は次の形式でも構いません。

1. Findings
2. UX issues
3. Visual issues
4. Suggested improvements
5. Short summary

## レビュー方針

- 最小変更を優先してください
- 問題点、運用リスク、拡張時の詰まりやすさを重視してください
- 実装を褒めることより、次に直す価値がある点を明確にしてください
- ファイルパスや対象箇所が分かる場合は明記してください
- デザインレビュー担当は、使いやすさ、情報の優先順位、最近の流行、効率、視認性、見た目の一貫性を重視してください
- デザインレビュー担当は、コードレビューと混ざらないよう `DESIGN_REVIEW.md` を使ってください

## 依頼時に明示してほしいこと

毎回、次の3点があるとレビュー往復が安定します。

- turn mode
  - `レビューのみ`
  - `記録反映あり`
  - `コード変更可`
- touched files
  - そのターンで触ったファイル一覧
- diff summary
  - `git diff --stat` 相当の短い要約

## 推奨テンプレート

```text
今回の依頼種別:
- <レビューのみ / 記録反映あり / コード変更可>

今回の差分:
- touched files: <ファイル一覧>
- diff summary: <短い要約>

レビュー対象:
- <見てほしい観点>

記録ルール:
- REVIEW.md は更新可 / 不可
- SHARE_NOTE.md は更新可 / 不可
- LOG.md は更新可 / 不可
```
